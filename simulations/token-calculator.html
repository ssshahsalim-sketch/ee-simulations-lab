05 // How Tokenization Works
A token is the smallest chunk of text a language model reads or writes — not always a whole word, and not always a single character. Models learn a fixed vocabulary of tens of thousands of these chunks, then break any input into the fewest pieces from that vocabulary. Common short words usually survive as one token; longer or unusual words often get split into pieces, called subword tokens. As a rough rule of thumb, Anthropic's own documentation notes that one token is roughly 4 characters or about three-quarters of a word in English.
Word count and token count drift apart fast. Short, common words often map to a single token, but longer, rarer, or compound words get sliced into pieces:
cat
≈ 1 token
tokenization
≈ 3–4 tokens
electromechanical
≈ 4–5 tokens
3,847,201
≈ 3–4 tokens
Numbers, code, and non-English scripts (like Hindi in Devanagari) typically use more tokens per character than plain English prose, because the model's vocabulary was trained mostly on English text. That's why a Hinglish WhatsApp-style message can cost noticeably more per word than an equivalent English sentence.
AI APIs bill per token, separately for what you send (input) and what the model generates (output) — output is typically priced several times higher than input. A system prompt, a long document, or a chatty back-and-forth all add up token by token, which is why the cost calculator above splits input and output and lets you project a daily workload into a monthly bill.
The pulse train above uses a simplified, rule-based splitting pattern built for teaching the concept — it is not the exact tokenizer used by Claude, GPT, or any other production model. Real tokenizers (usually Byte-Pair Encoding or similar) are trained statistically on huge text corpora, so their exact split points differ from this approximation. Treat the token count here as a reasonable ballpark for learning, not a billing-grade figure — for exact counts, use the model provider's official tokenizer or API usage response.
